CS 255 – DriverPass Portfolio Reflection

The DriverPass project focused on designing a web-based driver education system for DriverPass, a company seeking to improve the success rates of students on DMV driving tests. The client requested an integrated system that would support online practice exams, in person driving lessons and administrative reporting.
The solution was a browser-based system hosted in the cloud, supporting multiple user roles (student, instructor, secretary, management, and administrator) with role-based security, scheduling automation, DMV content updates, reporting features and secure data handling.
I believe that, after some clarification, I was able to clearly translate the needs of the client into a list of structured, testable requirements. My business requirement document was able to define the functional capabilities, nonfunctional requirements, and security constraints. Additionally, in the system design document, I was able to ensure that the technical requirements aligned with the clients’ expectations for cloud hosting, database management, scalability, and secure authentication. 

if there’s one area I’d revise, I would work to enhance the depth and clarity of my UML diagrams. While I believe they successfully represent the system interactions and entities they could be improved with the addition of clearer relationships, improved exception flows, and better labeling. This improvement would strengthen the connection between the requirements and system components and make the design documents much stronger.
Interpreting user needs requires careful analysis of the interview and translating the client’s informal statements into structured system requirements. For example, the customer concern about knowing who modified reservations became a requirement for audit logging and their desire to prevent double bookings became a scheduling conflict detection requirement. 
Considering user needs is important, as software systems exist to solve problems. A technically sound system that does not align with the stakeholder expectations and business workflows ultimately fails to achieve its initial purpose. Designing with the user in mind ensures the system is maintainable, scalable and supports long-term value for the business
My approach to designing software follows the following steps:
•	Identify the requirements
•	Categorize into functional / nonfunctional
•	Model objects, processes and technical specifications
Going forward, I’ll continue to use UML Diagrams, and work to understand and track the stakeholder’s requirements. As systems grow and increase in complexity, ensuring business goals and technical architecture align becomes critical. Combining documentation and modeling techniques provides clarity and scalability.
